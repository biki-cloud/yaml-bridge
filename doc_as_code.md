## どこが「ドキュメントのコード化」か

1. **ソースがコード（YAML）**
   - 要件・設計・実装メモなどはすべて `ai_document.yaml` という**構造化データ**で書かれている
   - スキーマ（`ai_document_scheme.json`）とガイド（`ai_document_guide.yaml`）で形式が決まっている

2. **ビルドで人間用ドキュメントを生成**
   - `create_human_document.py` が YAML を読み、Markdown（`human_document.md`）や Mermaid 図を**生成**している
   - コードをコンパイルして実行可能にするのと同様に、「YAML をビルドしてドキュメントを出す」流れになっている

3. **検証がある**
   - `validate.py` で YAML をスキーマ検証している
   - コードの Lint/テストのように、ドキュメントの「型」や整合性をチェックしている

4. **役割の分離**
   - **AI**：YAML を読み書きする（共通言語としての YAML）
   - **人**：ツールで生成された Markdown/図を見て確認する  
     という役割分担が `思想.md` の通り実装されている

## 一般的な「Documentation as Code」との対応

| 概念                              | このプロジェクトでの対応              |
| --------------------------------- | ------------------------------------- |
| ドキュメントをテキスト/構造で管理 | ✅ YAML（＋スキーマ）で管理           |
| バージョン管理                    | ✅ Git で管理可能                     |
| 自動検証                          | ✅ `validate.py`                      |
| ビルド/生成                       | ✅ `build.py` → Markdown/Mermaid 生成 |

なので、「ドキュメントをコード化している」という言い方で問題なく、`思想.md` の「YAML を介した共通言語」「ツールでマークダウンや Mermaid を作成」という設計が、そのまま「Documentation as Code」の形で実現されていると言えます。
